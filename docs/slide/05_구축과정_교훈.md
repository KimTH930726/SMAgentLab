# 슬라이드 5 — 구축하면서 알게 된 것

> 슬라이드 수: 2장
> 목표: 발표자가 직접 부딪힌 본질적인 질문들을 공유해 신뢰감을 높이고, 실무 AI 도입의 핵심을 전달한다
> 톤: "기술 문제" 나열이 아닌, "이 시스템을 왜 이렇게 만들었는가"의 관점

---

## [슬라이드 5-1] 만들면서 맞닥뜨린 본질적인 질문들

### 제목
**AI에게 우리 시스템을 이해시키는 것이 생각보다 어려웠다**

### 본문 내용

**① 맥락이 없으면 AI도 모른다**
- 처음엔 LLM API를 단순 호출 — 대화가 단건으로만 처리됨
- 맥락 유지를 위해 멀티턴 구조 구현 (대화 이력 + 요약 메모리)
- 용어집이 필요했던 이유: LLM이 우리 시스템 고유 약어·코드명을 모름
- **핵심 질문: "LLM에게 우리 시스템을 어떻게 이해시킬 것인가"**

**② 어떤 지식을, 어떤 형태로 쌓을 것인가**
- 운영 시스템 데이터를 어떤 단위로 쪼갤지 — 청킹 전략은 지금도 진행 중인 고민
- 시스템별·업무별로 **네임스페이스(파트)로 분리**해 검색 범위를 좁히는 방식을 선택
  - 현재는 이 네임스페이스가 필터링 구분 값 역할을 하는 수준
  - 지식을 카테고리·중요도로 세밀하게 구조화하는 단계까지는 아직 못 미침
- **한계점으로 남아 있는 것들**
  - 같은 네임스페이스 안에서 지식이 많아질수록 관련 없는 청크가 같이 검색되는 노이즈 문제
  - 청킹 단위가 너무 크면 핵심 내용이 묻히고, 너무 작으면 맥락이 끊겨 엉뚱한 답변이 나옴
  - 어떤 청킹 방식이 우리 운영 데이터에 최적인지 — 아직 정량적으로 검증된 기준이 없음
- **핵심 질문: "어떤 구조로 관리해야 우리 시스템에 가장 최적으로 검색될까"**

**③ AI가 혼자 판단하는 것을 어떻게 통제할 것인가**
- 벡터 검색에서 왜 이 문서가 나왔는지 보이지 않았음 → 파이프라인 디버그 구현
- 어떤 유사도로 어떤 청크가 검색됐는지, 어떤 퓨샷이 포함됐는지, 최종 LLM 프롬프트까지 투명하게 공개
- AI가 자율적으로 판단할수록 백오피스에서 통제·감사 기능이 필수적이라는 확신
- **핵심 질문: "AI의 판단 과정을 어떻게 가시화하고 통제할 것인가"**

### 발표자 노트
> "처음엔 그냥 LLM API 붙이면 되겠지 했는데, 실제로 해보니 AI가 우리 시스템을 이해하게 만드는 것 자체가 거의 반 이상의 작업이었습니다.
> 용어집을 만든 것도, 네임스페이스로 분리한 것도, 파이프라인 디버그를 만든 것도 전부 이 질문에서 나온 답들입니다.
> 특히 두 번째 질문 — 지식을 어떻게 쌓을 것인가 — 는 지금도 완전히 해결됐다고 말하기 어렵습니다.
> 현재는 네임스페이스로 검색 범위를 나누는 수준인데, 지식이 쌓일수록 같은 공간 안에서 노이즈가 생기고, 청킹 단위를 어떻게 잡느냐에 따라 품질이 크게 달라집니다.
> 이 부분은 앞으로도 계속 다듬어야 할 과제입니다."

## [슬라이드 5-2] 결국 핵심은 하나였다

### 제목
**AI 도입의 첫 단추 — 지식 파이프라인**

### 본문 내용

**이 프로젝트 전체를 관통하는 결론**

지식 파이프라인이 잘 구축되면 LLM 맥락 보강과 MCP·Agentic 확장까지 수월해진다. 하지만 파이프라인만큼 AI 판단에 대한 거버넌스와 투명성도 중요하다 — 여기서 얻은 세 가지 교훈은 다음과 같다.

| 교훈 | 내용 |
|------|------|
| **지식 파이프라인 먼저** | 우리 데이터를 손쉽게 AI가 이해할 수 있는 형태로 구축하고 현행화할 수 있어야 한다. 자동화와 진입 장벽이 핵심 |
| **맥락 보강이 모델 성능보다 중요** | 같은 LLM이어도 용어집·구조화된 지식·퓨샷이 있는 시스템이 없는 시스템보다 훨씬 낫다 |
| **거버넌스는 선택이 아니다** | AI가 자율화될수록 파이프라인 디버그·대시보드·쿼리 로그처럼 "AI가 왜 그 판단을 했는가"를 볼 수 있는 백오피스가 필수 |

**앞으로 남은 고민**
- 청킹 전략: 우리 운영 시스템 데이터에 최적화된 분할 방식은?
- 현행화 자동화: 지식이 outdated 되지 않도록 파이프라인을 어떻게 자동화할 것인가
- 최소 개입: 사람이 개입해야 하는 부분을 최소화하면서 품질을 유지하는 구조

### 발표자 노트
> "결국 AI 도입에서 가장 중요한 건 어떤 모델을 쓰느냐가 아니라, 우리 데이터를 얼마나 잘 AI에게 먹일 수 있느냐였습니다.
> 이게 해결되면 그다음 MCP나 에이전틱 확장은 의외로 수월합니다.
> 그리고 시스템이 자율화될수록 투명성과 통제 기능 — 백오피스 거버넌스 — 이 반드시 함께 가야 한다는 걸 이 프로젝트에서 배웠습니다."
