# 슬라이드 5 — 구축하면서 알게 된 것

> 슬라이드 수: 2장
> 목표: 발표자가 직접 부딪힌 본질적인 질문들을 공유해 신뢰감을 높이고, 실무 AI 도입의 핵심을 전달한다
> 톤: "기술 문제" 나열이 아닌, "이 시스템을 왜 이렇게 만들었는가"의 관점

---

## [슬라이드 5-1] 만들면서 맞닥뜨린 본질적인 질문들

### 제목
**AI에게 우리 시스템을 이해시키는 것이 생각보다 어려웠다**

### 본문 내용

| # | 질문 | 배운 것 |
|---|------|---------|
| ① | LLM에게 어떻게 이해시킬 것인가 | 개발 과정의 대부분을 차지한 어려운 작업 — 대화 맥락 유지(멀티턴+요약 메모리)와 사내 용어·약어를 이해시키는 용어집으로 대응 |
| ② | 어떤 청킹 전략·구조화를 거쳐야 하는가 | 지금도 연구 중인 개선 포인트 — 네임스페이스로 범위는 나눴지만, 지식이 쌓일수록 노이즈·청킹 단위 문제가 남아 있음 |
| ③ | 통제성·투명성을 어떻게 가시화할 것인가 | 자율성이 커질수록 반드시 필요 — 왜 이 문서가 검색됐는지 보이지 않아 파이프라인 디버그로 대응 |

### 발표자 노트
> "처음엔 그냥 LLM API 붙이면 되겠지 했는데, 실제로 해보니 AI가 우리 시스템을 이해하게 만드는 것 자체가 거의 반 이상의 작업이었습니다.
> 특히 두 번째 질문 — 지식을 어떻게 쌓을 것인가 — 는 지금도 완전히 해결됐다고 말하기 어렵습니다. 청킹 단위를 어떻게 잡느냐에 따라 품질이 크게 달라지는데, 아직 정량적으로 검증된 기준이 없어서 계속 다듬어야 할 과제입니다."

## [슬라이드 5-2] 결국 핵심은 하나였다

### 제목
**이 고민들이 풀려야 다음 확장이 열린다**

### 본문 내용

① **데이터가 모델보다 먼저다** — 같은 LLM이어도 용어집·구조화된 지식·퓨샷이 있는 시스템이 압도적으로 낫다
　　_자동화와 진입 장벽을 낮추는 것이 핵심_
② **거버넌스는 선택이 아니다** — 자율성이 커질수록 "왜 그 판단을 했는가"를 보여주는 백오피스가 필수
　　_투명성은 신뢰의 기반, 통제는 리스크 관리의 핵심_

→ 이 둘이 갖춰져야 MCP·에이전틱 같은 다음 확장이 수월해진다

**앞으로 남은 고민**
- 청킹 전략: 우리 운영 시스템 데이터에 최적화된 분할 방식은?
- 현행화 자동화: 지식이 outdated 되지 않도록 파이프라인을 어떻게 자동화할 것인가
- 최소 개입: 사람이 개입해야 하는 부분을 최소화하면서 품질을 유지하는 구조

### 발표자 노트
> "결국 AI 도입에서 가장 중요한 건 어떤 모델을 쓰느냐가 아니라, 우리 데이터를 얼마나 잘 AI에게 먹일 수 있느냐였습니다.
> 그리고 시스템이 자율화될수록 투명성과 통제 기능 — 백오피스 거버넌스 — 이 반드시 함께 가야 한다는 걸 배웠습니다.
> 이 둘이 갖춰지고 나면 그다음 MCP나 에이전틱 확장은 의외로 수월합니다."
