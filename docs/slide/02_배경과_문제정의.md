## [슬라이드 2-1] 기존 방식과 그 한계

### 제목
**우리가 지금까지 해온 방식 — 그리고 안 됐던 이유**

### 본문 내용

| 상황 | 기존 방식 | 문제점 |
|------|-----------|--------|
| 운영 이슈 발생 | 팀 내 숙련자에게 직접 물어봄 | 담당자 부재 시 대기, 지식이 개인에 종속 |
| 장애 대응 절차 확인 | 공유 드라이브/위키에서 직접 검색 | 문서 위치 찾는 데 시간 소요 |
| 반복 문의 처리 | 매번 같은 답변을 수동으로 작성 | 운영자 시간 낭비, 일관성 없는 답변 |
| 신규 팀원 온보딩 | 선배가 직접 설명 | 구두 전달 → 누락, 개인 편차 |

**기존 도구로는 왜 안 됐나**
1. **지식 레이어 부재** — 키워드 검색 수준에 그쳐, 의미로 관련 지식을 찾아주지 못함
2. **도메인을 모름** — 사내 시스템명·용어·약어("DS", "쿠폰 뺏어오기" 등)를 이해하지 못함
3. **품질 관리 불가** — 어떤 질문이 답을 못 찾는지, 어떤 지식이 문제인지 파악할 방법이 없음

### 핵심 메시지 (강조)
> **"지식이 사람과 개인 문서에 흩어져 있고, 기존 도구로는 이를 자산화·관리할 방법이 없었다"**

### 발표자 노트
> "운영 업무를 하다 보면 같은 질문이 반복되고, 그 답을 아는 사람이 자리를 비우거나 퇴직하면 지식이 사라집니다.
> 기존 위키나 검색 도구로도 안 되는 이유는 명확했습니다 — 의미 기반 검색이 없고, 사내 용어를 모르고, 품질을 관리할 방법이 없었습니다.
> 그래서 이 문제를 시스템으로 직접 해결하고 싶었습니다."

## [슬라이드 2-2] 해결하고자 한 니즈

### 제목
**시스템이 해결해야 할 것**

### 본문 내용

**사용자 관점**
- 언제든지 빠르게 운영 지식을 조회할 수 있어야 한다
- 자연어로 물으면 정확한 답변이 나와야 한다
- 이전 대화 맥락을 기억해야 한다

**운영자(지식 관리) 관점**
- 문서를 다양한 형식(Excel, PDF, Confluence 등)으로 등록할 수 있어야 한다
- 등록 후 검색 품질을 모니터링하고 개선할 수 있어야 한다
- 어떤 질문이 답을 못 찾는지 파악할 수 있어야 한다

**조직 관점**
- 특정 인물에 종속된 지식을 조직 자산으로 전환
- 운영 효율화 — 반복 업무를 시스템이 처리
- 지식이 쌓일수록 품질이 좋아지는 선순환 구조

### 발표자 노트
> "단순히 '검색이 되는 챗봇'이 아니라, 운영자가 지식을 관리하고 품질을 개선할 수 있는 구조가 필요했습니다.
> 그리고 시스템이 어느 부분에서 답을 못 찾는지 가시화할 수 있어야 한다고 생각했습니다."
