## [슬라이드 2-1] 기존 방식의 한계

### 제목
**우리가 지금까지 해온 방식**

### 본문 내용

| 상황 | 기존 방식 | 문제점 |
|------|-----------|--------|
| 운영 이슈 발생 | 팀 내 숙련자에게 직접 물어봄 | 담당자 부재 시 대기, 지식이 개인에 종속 |
| 장애 대응 절차 확인 | 공유 드라이브/위키에서 직접 검색 | 문서가 어디 있는지 찾는 데 시간 소요 |
| 반복 문의 처리 | 매번 같은 답변을 수동으로 작성 | 운영자 시간 낭비, 일관성 없는 답변 |
| 신규 팀원 온보딩 | 선배가 직접 설명 | 구두 전달 → 누락, 개인 편차 발생 |

### 핵심 메시지 (강조)
> **"지식이 사람 머릿속에만 있다 — 문서화되어도 찾기가 어렵다"**

### 발표자 노트
> "운영 업무를 하다 보면 같은 질문이 반복되는 경우가 많습니다.
> 그리고 그 답을 알고 있는 사람이 자리를 비웠을 때, 또는 퇴직했을 때 그 지식이 사라집니다.
> 이 문제를 시스템으로 해결하고 싶었습니다."

## [슬라이드 2-2] 해결하고자 한 니즈

### 제목
**시스템이 해결해야 할 것**

### 본문 내용

**사용자 관점**
- 언제든지 빠르게 운영 지식을 조회할 수 있어야 한다
- 자연어로 물으면 정확한 답변이 나와야 한다
- 이전 대화 맥락을 기억해야 한다

**운영자(지식 관리) 관점**
- 문서를 다양한 형식(Excel, PDF, Confluence 등)으로 등록할 수 있어야 한다
- 등록 후 검색 품질을 모니터링하고 개선할 수 있어야 한다
- 어떤 질문이 답을 못 찾는지 파악할 수 있어야 한다

**조직 관점**
- 특정 인물에 종속된 지식을 조직 자산으로 전환
- 운영 효율화 — 반복 업무를 시스템이 처리
- 지식이 쌓일수록 품질이 좋아지는 선순환 구조

### 발표자 노트
> "단순히 '검색이 되는 챗봇'이 아니라, 운영자가 지식을 관리하고 품질을 개선할 수 있는 구조가 필요했습니다.
> 그리고 시스템이 어느 부분에서 답을 못 찾는지 가시화할 수 있어야 한다고 생각했습니다."

## [슬라이드 2-3] 왜 직접 만들었는가

### 제목
**기존 솔루션으로 해결이 안 된 이유**

### 본문 내용

**폐쇄망 환경 제약**
- 외부 클라우드 AI 서비스(OpenAI, Google 등) 직접 연동 불가
- 사내 DevX LLM 게이트웨이(`devx-gw.shinsegae-inc.com`) 경유 필수
- 인터넷 없이 동작해야 함 → 오픈소스 임베딩 모델 직접 탑재

**도메인 특화 필요**
- 일반 LLM은 사내 시스템명, 운영 용어, 약어를 모름
- 조직 내 고유 용어를 인식하는 용어집(Glossary) 기능이 필요
- 사내 Confluence, 내부 문서 형식 지원 필요

**결론**
> "우리 환경에 맞는 시스템을 직접 설계할 수밖에 없었다"

### 발표자 노트
> "폐쇄망이라는 제약 조건이 있었고, 그 안에서도 LLM 품질을 최대한 활용하기 위해
> 임베딩 모델을 직접 컨테이너에 탑재하고, 사내 LLM 게이트웨이와 연동하는 구조를 선택했습니다."
