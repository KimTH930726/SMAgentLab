# 슬라이드 4 — 핵심 기술 구현 3가지

> 슬라이드 수: 3장 (각 기술 포인트 1장씩)
> 목표: 비기술 청중도 "왜 이렇게 만들었는지"를 납득할 수 있게 설명한다
> 방침: 코드/수식 최소화, 비유와 결과 중심으로

---

## [슬라이드 4-1] 지식 수집 자동화

### 제목
**무엇이든 지식으로 — 다양한 형식 지원**

### 본문 내용

**지원하는 지식 등록 방식**

| 방식 | 형식 | 특징 |
|------|------|------|
| 파일 업로드 | PDF, Excel(.xlsx), CSV, Markdown, TXT | 드래그앤드롭, 50MB 이하 |
| Confluence | 단일 페이지 / 하위 페이지 일괄 수집 | PAT 토큰 인증, BFS 트리 탐색 |
| URL | 일반 웹 페이지 | HTML 파싱 + 텍스트 추출 |
| 직접 입력 | 텍스트 직접 작성 | — |

**등록 후 자동으로 처리되는 것들**
- LLM이 문서를 읽고 카테고리·키워드 자동 태깅
- 사내 고유 용어를 자동 추출해 용어집 후보 생성
- 유사 Q&A 예제 자동 생성 (Few-shot 후보)
- 적절한 크기로 청크 분할 + 임베딩 벡터 생성

**Human-in-the-loop**
- 등록 전 청크 단위 미리보기 → 불필요한 내용 선택 해제 후 등록

### 발표자 노트
> "Excel이나 PDF 같은 기존 업무 문서를 그대로 올리면 시스템이 알아서 쪼개고, 의미를 벡터로 변환해서 저장합니다.
> 등록자가 원하면 어떻게 쪼갰는지 미리 보고 수정할 수도 있습니다."

## [슬라이드 4-2] 도메인 특화 기능 — 이 시스템만의 차별점

### 제목
**일반 RAG와 다른 것들**

### 본문 내용

**① 용어집 (Glossary) — 사내 언어를 시스템이 이해한다**

```
사용자 질문: "쿠폰 DS 적용 안 됨"
                    ↓ 용어집 유사도 검색
용어집 히트: "DS" = "Discount Service (할인 정책 엔진)"
                    ↓ 검색어 자동 보강
실제 검색어: "쿠폰 DS 적용 안 됨  +  Discount Service"
```
- 운영자가 사내 고유 용어 ↔ 정식 명칭을 직접 등록
- 문서에 없어도 약어/코드명으로 질문하면 자동 매핑

**② Q&A 예제 (Few-shot) — 정답 패턴을 미리 학습시킨다**
- 관리자가 "이런 질문엔 이렇게 답해야 한다"는 예제를 등록
- 유사 질문이 들어오면 해당 예제를 LLM 컨텍스트에 자동 포함
- 답변 톤·형식까지 제어 가능

**③ 파이프라인 디버그 — 왜 이 답이 나왔는지 투명하게**

```
[어드민 디버그 패널]
  질문 입력 → 파트 선택 → 용어 매핑 → 지식 검색
  → Q&A → MCP 도구 → LLM 컨텍스트 → 결과
  
  각 단계별 유사도 점수 / 신뢰도 / 매핑된 용어 확인 가능
  → "왜 이 문서가 검색됐는가"를 숫자로 확인
```

**④ MCP 도구 호출 — 단순 Q&A를 넘어 실행까지**
- 외부 HTTP API를 "도구"로 등록하면 LLM이 필요 시 자동 호출
- 예: 특정 시스템 상태 조회 API → 질문에서 자동 호출 → 실시간 결과 답변에 포함
- 사람이 API 문서 찾아보지 않아도 됨

### 발표자 노트
> "하이브리드 검색 자체는 이제 대부분의 RAG에서 채택하는 방식입니다.
> 이 시스템이 다른 건, 사내 언어를 이해하는 용어집, 답변 패턴을 학습시키는 Q&A 예제,
> 그리고 파이프라인 전 단계를 점수로 투명하게 보여주는 디버그 기능입니다.
> 특히 MCP 도구 연동은 단순 지식 조회를 넘어 실제 시스템을 건드릴 수 있다는 차별점입니다."

## [슬라이드 4-3] 지속 개선 루프 — 쓸수록 좋아지는 구조

### 제목
**한 번 구축하고 끝이 아닌 이유**

### 본문 내용

**개선 루프 구조**

```
사용자 질문
    │
    ├─ 피드백 (👍/👎)
    │       └─▶ 해당 지식의 검색 가중치(base_weight) 자동 조정
    │              좋은 답변 → 가중치 ↑ / 나쁜 답변 → 가중치 ↓
    │
    ├─ 지식 갭 감지
    │       └─▶ 관련 지식을 못 찾은 질문 → DB에 "no_knowledge" 기록
    │              어드민에서 "아직 등록 안 된 주제" 파악 가능
    │
    └─ 대화 메모리
            └─▶ 4번 대화마다 LLM이 대화를 요약 → 벡터로 저장
                   이후 대화에서 과거 맥락 자동 리콜
```

**대시보드 — 현황을 한눈에, 조치까지 원클릭**

```
[어드민 대시보드 — KPI 카드 5개 + 도넛 차트]

  전체  |  해결됨  |  대기중  |  미해결  |  ⚠ 지식 공백
                                           └─ 건수 있으면 주황 강조
                                              "등록 필요" 뱃지 표시
```

```
[도넛 차트 세그먼트]
  ● 초록 — 해결됨   ● 노랑 — 대기중
  ● 빨강 — 미해결   ● 주황 — 지식 공백 (등록된 지식이 아예 없었던 질문)
```

- **⚠ 지식 공백 카드** 클릭 → 해당 질문 목록 모달
  → 각 항목 클릭 → 지식 등록 폼 자동 팝업 (질문이 원본으로 표시)
  → 내용 작성 후 등록 → 즉시 검색에 반영 + "해결됨" 처리

**왜 이게 중요한가**
- 피드백이 쌓일수록 → 좋은 지식이 위로, 나쁜 지식은 아래로
- 갭 감지 기록이 쌓일수록 → 무엇을 추가 등록해야 할지 명확해짐
- 대시보드에서 확인 → 원클릭 등록 → 즉시 반영의 관리 사이클 완성
- 운영자가 최소한의 노력으로 품질을 지속적으로 높일 수 있음

### 발표자 노트
> "이 구조가 '지속 가능한 RAG'라는 제목의 핵심입니다.
> 사용자가 그냥 쓰기만 해도, 피드백 하나하나가 시스템을 개선하는 신호가 됩니다.
> 특히 대시보드에서 '지식이 없어서 답 못 한 질문'을 클릭하면,
> AI가 그때 생성했던 답변이 자동으로 폼에 채워집니다.
> 수정해서 등록하면 바로 검색에 반영되고, 해당 질문은 '해결됨'으로 처리됩니다.
> 관리자가 모니터링하면서 지식을 채워나가는 사이클이 대시보드 안에서 완결됩니다."
