# 슬라이드 4 — 핵심 기술 구현 3가지

> 슬라이드 수: 3장 (핵심 메시지 3개 = 1장씩)
> 목표: "일반 RAG와 뭐가 다른가"에 대한 답을 정확히 3가지로 압축해 전달한다
> 방침: 코드/수식 최소화, 비유와 결과 중심으로. 파이프라인 세부 구조는 다루지 않는다
> 제목 규칙: 큰 제목에 "차별점 ①/②/③ / 3"을 넣어 지금 3가지 중 몇 번째인지 한눈에 보이게 한다

## [슬라이드 4-1] 차별점 ① — 등록부터 이해까지

### 제목
**차별점 ① / 3 — 등록부터 이해까지**

### 본문 내용

**지원하는 지식 등록 방식**

| 방식 | 형식 | 특징 |
|------|------|------|
| 파일 업로드 | PDF, Excel(.xlsx), CSV, Markdown, TXT | 드래그앤드롭, 50MB 이하 |
| URL / Confluence | 일반 웹 페이지 · Confluence 단일/하위 페이지 일괄 수집 | URL 패턴으로 자동 구분, Confluence는 PAT 인증 + BFS 트리 탐색 |
| 직접 입력 (단건) | 텍스트 직접 작성 | 지식 1건으로 즉시 등록 |
| 대량 텍스트 | 긴 텍스트 붙여넣기 | 단락/고정 길이 기준으로 분할 → 여러 청크로 등록 |

등록 시 미리보기에서 청크 단위로 확인하고, 관리자가 최종 확정합니다(Human-in-the-loop).

**등록만으로 끝나지 않는다 — 용어집·퓨샷으로 지식을 보강**

- **용어집(Glossary)** — 사내 고유 용어·약어를 정식 명칭과 매핑. "DS" → "Discount Service"처럼 문서에 없는 표현으로 질문해도 자동 보강되어 검색됨
- **Q&A 예제(Few-shot)** — "이런 질문엔 이렇게 답해야 한다"는 예제를 등록하면, 유사 질문 시 LLM 컨텍스트에 자동 포함 — 답변 톤·형식까지 제어 가능

### 발표자 노트
> "지식을 등록하는 문턱을 최대한 낮췄습니다 — 파일이든 URL이든 텍스트든 같은 파이프라인을 탑니다.
> 그런데 등록만으로는 부족합니다. LLM은 우리 회사 용어를 모르기 때문에, 용어집과 Q&A 예제로 등록된 지식을 실제로 '이해할 수 있는 형태'로 보강하는 게 이 시스템의 첫 번째 차별점입니다."

## [슬라이드 4-2] 차별점 ② — 선순환 구조

### 제목
**차별점 ② / 3 — 선순환 구조**

### 본문 내용

```
사용자 질문
    │
    ├─ 피드백 (👍/👎)
    │       └─▶ 해당 지식의 검색 가중치(base_weight) 자동 조정
    │              좋은 답변 → 가중치 ↑ / 나쁜 답변 → 가중치 ↓
    │
    └─ 지식 공백 감지
            └─▶ 관련 지식을 못 찾은 질문 → DB에 "no_knowledge" 기록
                   운영자가 "아직 등록 안 된 주제"를 파악할 수 있음
```

**왜 이게 중요한가**
- 사용자가 그냥 쓰기만 해도, 피드백 하나하나가 지식 품질을 조정하는 신호가 됨
- 좋은 지식은 위로, 나쁜 지식은 아래로 — 별도 관리 없이도 검색 품질이 스스로 개선됨
- "몰라서 답 못 한 질문"이 기록으로 남아, 다음에 뭘 채워야 할지가 명확해짐

### 발표자 노트
> "이 선순환 구조가 '지속 가능한 RAG'라는 제목의 핵심입니다.
> 사용자는 그냥 평소처럼 질문하고 피드백을 누르기만 하면 되는데, 그 행동 하나하나가 시스템을 스스로 개선시키는 신호로 쓰입니다."

## [슬라이드 4-3] 차별점 ③ — 대시보드

### 제목
**차별점 ③ / 3 — 대시보드**

### 본문 내용

```
[어드민 대시보드 — KPI 카드 5개 + 도넛 차트로 상태 표시]
  ● 해결됨(초록)  ● 대기중(노랑)  ● 미해결(빨강)
  ● ⚠ 지식 공백(주황) — 등록된 지식이 아예 없었던 질문. 건수 있으면 강조 + "등록 필요" 뱃지 표시
```

- **⚠ 지식 공백 카드** 클릭 → 해당 질문 목록 모달
  → 각 항목 클릭 → 지식 등록 폼 자동 팝업 (질문이 원본으로 표시)
  → 내용 작성 후 등록 → 즉시 검색에 반영 + "해결됨" 처리

관리자가 검색 과정 자체를 들여다보고 싶을 땐, 파이프라인 디버그 패널에서 어떤 문서가 어떤 유사도로 검색됐는지 단계별로 확인할 수 있습니다.

**왜 이게 중요한가**
- 지식 공백 기록이 쌓일수록 → 무엇을 추가 등록해야 할지 명확해짐
- 대시보드에서 확인 → 원클릭 등록 → 즉시 반영의 관리 사이클 완성
- 운영자가 최소한의 노력으로 품질을 지속적으로 높일 수 있음

### 발표자 노트
> "지식 공백이 어디에 있는지 대시보드에서 클릭 한 번이면 확인할 수 있고, 그 자리에서 바로 등록까지 이어집니다.
> 왜 이 문서가 검색됐는지 궁금하면 디버그 패널에서 단계별 점수까지 확인할 수 있어요 — 결과와 과정을 둘 다 투명하게 볼 수 있게 만든 게 핵심입니다."
